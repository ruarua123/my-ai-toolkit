# G-Plan 计划命令

本命令用于在编写任何代码之前，创建一份完整的实施计划。

## 功能说明

1. **重述需求** — 用清晰的术语重新说明要构建什么
2. **识别风险** — 发现潜在问题和阻塞项
3. **创建分步计划** — 将实施过程拆分为多个阶段
4. **等待确认** — 必须收到用户明确同意后才开始编码

## 何时使用

在以下场景中使用 `/g-plan`：
- 开始一个新功能
- 进行重大架构变更
- 复杂的重构工作
- 涉及多个文件/组件的改动
- 需求不清晰或存在歧义

## 工作流程

### 第一步：分析需求

仔细阅读用户的请求 `$ARGUMENTS`，然后：

1. **用你自己的话重述需求**，确保理解正确
2. **列出关键约束条件**（技术栈、性能要求、兼容性等）
3. **明确完成标准**（怎样算"做完了"）

### 第二步：探索代码库

- 阅读相关的源代码文件
- 理解当前的架构和模式
- 识别需要修改或新增的文件
- 了解已有的测试覆盖情况

### 第三步：制定计划

将实施方案拆分为多个阶段，每个阶段包含具体的、可执行的步骤。

**计划格式要求：**

```
# 实施计划：[功能名称]

## 需求重述
- （用清晰的语言描述要构建什么）
- （关键约束条件）
- （完成标准）

## 实施阶段

### 阶段 1：[阶段名称]
- [ ] 步骤 1：具体描述
- [ ] 步骤 2：具体描述
- **涉及文件**：file1.ts, file2.ts

### 阶段 2：[阶段名称]
- [ ] 步骤 1：具体描述
- [ ] 步骤 2：具体描述
- **涉及文件**：file3.ts, file4.ts

（更多阶段...）

## 依赖关系
- （各阶段/组件之间的依赖说明）
- （需要的外部服务或库）

## 风险评估
- 🔴 高风险：（描述及缓解措施）
- 🟡 中风险：（描述及缓解措施）
- 🟢 低风险：（描述及缓解措施）

## 复杂度评估
- 整体复杂度：高/中/低
- 预计影响范围：X 个文件
```

### 第四步：等待确认

**关键规则**：在用户明确确认之前，**不要写任何代码**。

计划完成后，输出：

```
📋 计划已制定完成。

请确认是否按此计划执行：
- 回复「确认」或「开始」→ 按计划执行
- 回复「修改：xxx」→ 调整计划中指定的部分
- 回复「换方案：xxx」→ 提出替代方案
- 回复「跳到阶段 X」→ 跳过前面的阶段
```

## 重要原则

1. **先理解，再行动** — 不要急于动手，先确保理解正确
2. **分而治之** — 大任务拆成小步骤，每个步骤都要具体可执行
3. **识别依赖** — 明确哪些步骤依赖其他步骤的完成
4. **评估风险** — 提前发现可能的问题，并给出缓解措施
5. **尊重确认** — 用户没有点头之前，一行代码都不要动

## 后续操作

计划确认并执行后，建议配合以下命令使用：
- `/code-review` — 审查完成的实现
- `/tdd` — 使用测试驱动开发方式实现
- `/refactor` — 代码优化和重构

## 示例

```
用户：/g-plan 给项目添加用户认证功能，支持邮箱和手机号登录

Agent：
# 实施计划：用户认证功能

## 需求重述
- 为项目添加完整的用户认证系统
- 支持两种登录方式：邮箱登录 + 手机号登录
- 需要包含注册、登录、登出、密码重置功能
- 完成标准：用户可以通过邮箱或手机号完成注册和登录

## 实施阶段

### 阶段 1：数据库设计
- [ ] 创建 users 表（id, email, phone, password_hash, created_at）
- [ ] 创建 user_sessions 表（token, user_id, expires_at）
- [ ] 添加必要的索引（email 唯一索引、phone 唯一索引）
- **涉及文件**：migrations/001_create_users.sql

### 阶段 2：认证服务
- [ ] 实现邮箱注册逻辑（含邮箱格式验证）
- [ ] 实现手机号注册逻辑（含验证码发送）
- [ ] 实现 JWT token 生成与验证
- [ ] 实现密码哈希（bcrypt）
- **涉及文件**：src/auth/service.ts, src/auth/utils.ts

### 阶段 3：API 接口
- [ ] POST /api/auth/register — 注册
- [ ] POST /api/auth/login — 登录
- [ ] POST /api/auth/logout — 登出
- [ ] POST /api/auth/forgot-password — 忘记密码
- [ ] POST /api/auth/reset-password — 重置密码
- **涉及文件**：src/auth/routes.ts, src/auth/controllers.ts

### 阶段 4：前端页面
- [ ] 创建注册页面
- [ ] 创建登录页面
- [ ] 添加认证状态管理
- [ ] 实现路由守卫（未登录跳转登录页）
- **涉及文件**：src/pages/auth/, src/middleware/auth.ts

## 依赖关系
- 阶段 2 依赖阶段 1（数据库表必须先创建）
- 阶段 3 依赖阶段 2（服务层必须先实现）
- 阶段 4 依赖阶段 3（API 接口必须先就绪）
- 外部依赖：短信服务商 API、邮箱服务（SMTP/SES）

## 风险评估
- 🔴 高风险：短信验证码发送的安全性（需防刷、限流）
- 🟡 中风险：JWT token 过期与刷新策略
- 🟡 中风险：密码重置流程的安全性
- 🟢 低风险：邮箱格式验证

## 复杂度评估
- 整体复杂度：中高
- 预计影响范围：约 12-15 个文件

📋 计划已制定完成。

请确认是否按此计划执行：
- 回复「确认」或「开始」→ 按计划执行
- 回复「修改：xxx」→ 调整计划中指定的部分
- 回复「换方案：xxx」→ 提出替代方案
- 回复「跳到阶段 X」→ 跳过前面的阶段
```
